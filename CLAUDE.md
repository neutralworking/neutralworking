# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Repo shape

Pure static site served at `neutralworking.com`. No package.json, no build step, no test suite. Everything is hand-written HTML/CSS/JS in single self-contained files — there is no bundler, no framework, no dependency graph. To preview, open the file in a browser or run `python3 -m http.server` from the repo root.

Four top-level surfaces, all linked from `index.html`:

1. **`index.html`** — landing page for "Neutral Working" (Luke Warrington). Single file, inline CSS/JS. Renders as a VS Code window (titlebar, tab strip, explorer, editor, status bar) with a syntax-highlighted TypeScript hero.
2. **`admin/index.html`** — ops dashboard (portfolio health, KB freshness, Studio queue, script runner) behind a shared-secret auth gate. Shares the landing's VS Code aesthetic.
3. **`studio/`** — "Language Tidbits Studio", a single-file SPA (`studio/index.html`, ~875 lines).
4. **`kb/`** — static mirror of the "Chief Scout Knowledge Base".

## `studio/` — how it works

`studio/index.html` is one self-contained file with three panels (submit → progress → review) that talk to an **n8n** backend. The frontend only knows the webhook base path:

```js
const WEBHOOK_BASE = "/n8n/webhook";
```

Endpoints it expects (not in this repo — lives in the n8n instance):
- `POST /studio/submit` → `{ input }` returns `{ id }`
- `GET  /studio/topics` → list of topics
- `GET  /studio/status?id=…` → `{ status, stages: { ingest, summarize, tts, render, ready } }`, polled every 3s
- `POST /studio/approve` → `{ id, platforms, title, description, tags }`
- `POST /studio/reject`  → `{ id, reason }`

Pipeline stage keys (order matters, used for DOM selectors): `ingest`, `summarize`, `tts`, `render`, `ready`. If you add a stage, update both `STAGE_KEYS` and the `[data-stage="…"]` elements in the markup.

Studio uses a different palette from the rest of the site (`#005FFF` accent, Helvetica, 13px base) — don't "unify" it with the landing/kb styling unless asked.

## `kb/` — static mirror, do not hand-edit

`kb/` is a **generated** static mirror of the Chief Scout knowledge base. The header of `kb/INDEX.md` says it all:

> Auto-generated by `pipeline/96_kb_index.py` — do not edit manually.

The generator (`pipeline/96_kb_index.py`) is **not in this repo** — it lives in the upstream Chief Scout project and the built output is committed here. Each article exists as a pair: `foo.md` (source with YAML frontmatter: `title, category, tags, updated, source, confidence, summary`) and `foo.html` (rendered sibling with topnav + `view raw .md →` link back to the `.md`). Categories under `kb/`: `archetypes/`, `clubs/`, `concepts/`, `players/`, `queries/`, `tactics/`. Each category has an `_index.md` + `index.html`. Currently only `archetypes` (13) and `tactics` (33) have content; the others are scaffolded empty.

Rules when working in `kb/`:
- **Do not hand-edit `.md` or `.html` files to fix content** — changes belong in the upstream pipeline and will be overwritten on the next rebuild. Acceptable edits here are limited to the presentation shell (shared CSS/topnav) if the user explicitly asks.
- If asked to change an article's *content*, flag that the source is upstream before editing.
- Wikilinks render as `<span class="wikilink-broken">slug</span>` when the target article doesn't exist yet — this is expected for players/clubs/etc. not mentioned above.

## `admin/` — how it works

`admin/index.html` is a single self-contained dashboard that shares `index.html`'s VS Code aesthetic (same tokens, same window chrome) and talks to the same n8n instance as `studio/` but under a separate path:

```js
const WEBHOOK_BASE = "/n8n/webhook/admin";
```

**Auth**: a shared secret stored in `localStorage` under `nw.admin.token`, sent as the `X-Admin-Token` header on every fetch. The auth modal is purely client-side — **real enforcement must live in n8n**. Anyone who reads the JS sees the header name, and the token is visible in the network tab on every request. If this page is ever served without n8n validating the header, it's wide open. There is a prominent JSDoc block at the top of the `<script>` repeating this.

**Mock mode**: on by default (`localStorage.nw.admin.mock !== "0"`). Every endpoint is shimmed by an in-file `MOCK` object and a stateful mock job engine that simulates log streaming. A pill in the titlebar and the status-bar band color (red = mock, green = live) indicate the current mode. This lets the UI ship and be reviewed before any n8n workflow exists.

**Endpoint contract** (all under `/n8n/webhook/admin/`, all carry `X-Admin-Token`):

| Method | Path                        | Purpose                                     |
|--------|-----------------------------|---------------------------------------------|
| GET    | `/portfolio`                | uptime rows with `history[]` for sparklines |
| POST   | `/portfolio/recheck`        | force re-run of all checks                  |
| GET    | `/kb/status`                | last rebuild + category counts + stale list |
| POST   | `/kb/rebuild`               | kick off `pipeline/96_kb_index.py`, returns `job_id` |
| GET    | `/studio/queue`             | topics in flight, with 5-stage state        |
| POST   | `/studio/rerun`             | `{ id }` → `job_id`                         |
| GET    | `/workflows`                | script manifest: `[{ id, label, desc, params }]` |
| POST   | `/scripts/run`              | `{ id, params }` → `{ job_id }`             |
| GET    | `/scripts/status?job_id=…`  | `{ state, logs, exit_code }` (polled every 1.5s) |

If any endpoint returns 404 in live mode, the owning panel renders a dashed "not wired" empty state showing the exact path it tried. The page never breaks on a missing backend.

The 5 Studio pipeline stage keys (`ingest`, `summarize`, `tts`, `render`, `ready`) mirror `studio/index.html`'s `STAGE_KEYS` exactly — keep them in sync if either file changes.

## Styling conventions

Landing (`index.html`) and Admin (`admin/index.html`) share a **VS Code High Contrast Dark** visual language: pure black bg, bright colored accents (`--kw` `#4FC1FF`, `--type` `#4EC9B0`, `--fn` `#DCDCAA`, `--str` `#CE9178`, `--err` `#F48771`, `--cmt` `#7CAE6C`), fixed 34px titlebar / 32px tab strip / 24px status bar chrome, `Geist Mono` as the primary font with `Instrument Serif` for one hero accent on the landing, scanline + SVG grain overlays. The full token block is duplicated at the top of both files — if you adjust tokens, update both.

KB (`kb/**/*.html`) uses a **different, older** visual language (JetBrains Mono, `#0a0a0a` bg, `#00ff88` accent, grid background via `body::before`). It is a generated mirror (see below) and should not be restyled to match the new landing unless you're prepared to rewrite the upstream generator.

Studio (`studio/index.html`) uses a **third, deliberately distinct** palette (`#005FFF` accent, Helvetica, 13px base). Don't unify it with landing/admin unless asked.

All CSS and JS is inlined per-file. There is no shared stylesheet — if you change the landing/admin token palette, the update must be hand-propagated. Same for KB chrome, preferably via the upstream generator.
