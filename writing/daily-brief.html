Founder/PO Daily Brief — dogfood first

draft · 2026-05-16 · ~6 min read

The hardest part of selling "I'll build an AI ops layer for your team" is having a manual ops layer yourself. So before pitching the AI Operations Setup service to anyone, I built the workflow I'd want for me. This is the write-up.

Two outputs: a daily brief at 07:00 UTC and a weekly brief on Sunday evening. Both arrive on my phone before I open a laptop. The daily one is opinionated — it picks the single most important thing I should do today and tells me to do it. The weekly one is reflective — what shipped, what's blocked, what needs attention, what's next.

Why this comes first

  • Demo for sales. A working brief delivered to my phone is a more honest pitch than a slide deck.
  • Personal productivity. The portfolio is too wide to hold in my head — Chief Scout, Kickoff Clash, Crumble, Neutral Working all compete for attention.
  • Reusable template. The same skeleton (ingest → summarise → deliver) is the foundation of the AI Ops System offer.
  • Dogfood credibility. If I'm not running it on myself I shouldn't sell it.

Each source is a separate n8n credential and a separate node. If one fails, the brief still ships — it just notes the missing section. Better a partial brief than no brief.

Outputs

Daily brief (07:00 UTC, Telegram)

  • Today's agenda + a one-line prep note for each meeting
  • Outstanding follow-ups (you said you'd reply by today)
  • Overdue tasks across all projects
  • Suggested focus block — the one thing that moves things forward most
  • Cross-project risks / blockers

Weekly brief (Sunday 18:00 UTC, Email)

  • What shipped this week (git log across active repos + closed Linear issues)
  • What's blocked and why
  • What needs attention before Monday
  • Next week's two priorities

Workflow shape

# one n8n workflow, two triggers (daily 07:00 / weekly Sun 18:00)

cron → fan-out parallel ingest:
  ├─ gmail        (label:@brief, starred)
  ├─ calendar     (today + tomorrow)
  ├─ linear       (assignee:me)
  ├─ github       (open prs + mentions)
  └─ drafts       (48h window)

→ normalise to {source, ts, kind, title, snippet, url}
→ rank by importance (claude haiku, cheap pass)
→ summarise top N per section (claude opus)
→ pick one_thing (heuristic + claude tie-break)
→ render markdown
→ deliver (telegram or email)
→ log to postgres decisions(id, kind, payload, delivered_at)

Total runtime budget: 10 seconds. Anything slower than that and I'd rather skip a section than push past the wake-up window.

Rules I gave myself

  • Read-only on v1. No sending emails on my behalf. No auto-creating tickets. Suggest, don't act.
  • Single user. Multi-account support is a productisation problem, not a dogfood one.
  • Partial > none. If Linear is down, the brief ships without that section and flags it.
  • Decisions are logged. Every brief and every "one thing" pick gets a row in Postgres so I can audit it.
  • Boring observability. The admin panel shows the last 6 runs, source health, and a one-click rerun. Nothing more.
// out of scope for v1 Sending mail on my behalf. Creating tickets automatically. Multi-account / multi-user. Slack ingest is wired but not connected — added in v2 once I prove the daily one is useful.

What this unlocks

Once the daily brief is stable, the same skeleton becomes:

  • Meeting-to-Tasks Bot — same ingest/summarise/deliver shape, narrower input (transcript) and richer output (Linear tickets). That's NW-34.
  • AI Operations Setup service page — the productised offer pitched on the back of this dogfood (NW-18).

In other words: the brief isn't the product. The brief is the proof I can build the product.
