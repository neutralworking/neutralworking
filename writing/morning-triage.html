Morning triage — keeping the backlog honest

draft · 2026-05-19 · ~5 min read

The morning routine I run on my own product backlog. Pull yesterday's standups, grill the tracker into honest state, push corrections back, draft today's standup. Forty-five minutes on a clean week, closer to ninety when the table has drifted.

Same logic as the daily brief: dogfood before pitching. That piece was the AI ops layer I'd want for me. This one is the Fractional PO discipline I'd want a Fractional PO to have.

Why this comes first

  • Trackers lie if you don't ask them. Sprint tools accumulate inaccuracy unless someone walks them every day. Statuses go stale, "blocked on X" turns into "blocked on me writing X a message," "Live" hides three follow-up bugs.
  • The discipline is the offer. "I'll keep your product backlog honest" is a stronger pitch than "I'll add another tool." A Fractional PO doesn't earn their fee on the days nothing's broken — they earn it on the days the table was about to lie to someone and didn't.
  • Reusable across clients. The four-stage shape is portable. Different tracker, different tickets, same pattern.

The four stages

# roughly 45 minutes, every weekday

1. pull     yesterday's standups + any new asks into the conversation
2. grill    each row that moved — surface lies, lazy statuses, hidden blocks
3. push     corrections back into the tracker (status, priority, notes, splits)
4. draft    today's standup from the corrected state, not from memory

The order matters. Drafting the standup before the grill is how you end up writing yesterday's lies again. The standup is the artefact — the grill is what makes it true.

The grill

The most useful step. Three failure modes I look for, in order of frequency:

"Blocked" that isn't

Most "blockers" are procrastinated tasks pretending to be external dependencies. "Blocked on X approving the access" → actually, "blocked on me writing X a message asking for the access." The table lets you hide behind that for weeks if you let it.

The grill question: what does the next move look like, and whose desk does it sit on? If the answer is "mine, but I haven't done it," the status isn't Blocked, it's avoidance.

"Live" that isn't

Tickets get marked Live when the code ships. But a feature that's deployed and gated behind a Marketing email that hasn't gone out yet isn't live — it's staged. A feature shipped with a known cosmetic bug isn't done — it's released with debt. Both deserve their own state. The status vocabulary I've settled on:

StatusMeans
TestingQA in flight, not yet on prod
Ready to ReleaseOn prod, gated on an external trigger (email send, embargo, dependency)
LiveUsers have it, no follow-up trigger waiting
DoneClosed. No further action. Ticket goes to bed.

"Ready to Release" is the one most boards are missing. It's where a lot of half-shipped work lives, mislabelled as Live.

Priority drift

Stars haven't been reviewed since the work was scoped. The upstream context has moved — a dependent feature shipped, a stakeholder lost interest, a competitor changed pricing — but the priority on the row still says ⭐⭐⭐⭐⭐. A priority that hasn't been re-justified this week is a priority you don't trust.

What the tracker holds

Whatever the tool, the columns are the same:

FieldWhy
Ticket IDThe canonical reference. Separate field, not jammed into the title.
TitlePlain-English description. Should make sense without the ID.
ProjectOne-of, so cards group cleanly on a Kanban.
TypeFeature / Bug / QA / Ops / Research / Chore. Lets you filter "all bugs this week."
StatusFrom the list above. Honest, not aspirational.
PriorityRe-justified weekly, not anchored to the scoping moment.
Status NotesOne-paragraph latest state. Append-don't-rewrite is a rule.
NotesLong-form context: PRDs, links to source documents, history.
URLPointer to the canonical ticket. Saves a search every time someone wants the source.

Views matter as much as fields. The minimum useful set:

  • Kanban (Active) — group by Status, hide On Hold + Done. The day-to-day view.
  • This week — Testing / Ready to Release / Development / Blocked, sorted by priority. The morning view.
  • Blocked & waiting — forces visibility of every external dependency. The audit view.
  • Shipped recently — filtered to Done in the last 30 days. The standup-summary view.

Rules I gave myself

  • Don't move a row without honest reasoning. If you can't say why the status changed in one sentence, the move isn't done.
  • Append, don't rewrite. Status Notes is a log of state, not a fresh paragraph each morning.
  • Splits over stuffing. If a row covers two pieces of work — feature + follow-up bug, design + implementation — split them. One ticket per addressable action.
  • Closed should mean closed. If the ticket can be re-opened in a week, it isn't Done. Use a follow-up row.
  • Read-only on automation, for now. The grill is a human pattern. A model can draft notes and surface mismatches, but the call to move a row stays with the human.
// out of scope for v1 Auto-creating or auto-closing tickets. Auto-replying on review threads. Replacing the grill with a model. Reading other people's backlogs. The output is honest state — that doesn't require automation, it requires showing up.

What this unlocks

Once the morning routine is stable, the same pattern becomes:

  • A retainable client offer — Fractional PO at a fixed weekly cadence, where the morning triage is the core service and everything else (roadmap, sprint planning, stakeholder management) sits on top of an honest tracker.
  • A portable template — same tracker shape, same view set, same status vocabulary across clients. New engagement, one hour to stand the table up.
  • A weekly artefact — the "shipped recently" view + the "blocked & waiting" view, sent to whoever needs visibility. Less effort than a status email and more useful.

The brief isn't the product. The morning routine isn't the product either. The product is the table not lying when someone looks at it.
